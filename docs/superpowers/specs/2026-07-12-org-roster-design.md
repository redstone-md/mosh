# Organization Roster — Design

Date: 2026-07-12
Status: draft for review

## Goal

Let a team of 5–30 people onboard into Mosh with a single artifact and let an
org admin revoke a member's access with one action. Today onboarding is
pairwise invite URIs and revocation does not exist. This is the feature gap
between "hobby messenger" and "deployable for a team".

An **organization** is a signed membership document (roster), not a server.
No new infrastructure is introduced; the zero-server property is preserved.

## Non-goals (v1)

- Per-member identity key chains (org cert → member key → MLS credential).
  v1 pins the stable moss peer-id per member; the key chain is the v2 upgrade
  if peer-id pinning proves insufficient.
- Admin GUI. v1 admin tooling is a CLI in a separate repository (closed
  source); this repo only contains the verification/consumption side.
- Mobile clients, multi-device per member.
- Roster-driven policy beyond membership (retention, export) — later.

## Trust model

- **Org root key**: Ed25519 keypair generated by the admin CLI. Private key
  never enters this codebase. The org's identity *is* this public key.
- **Roster**: canonical JSON document signed by the org root key:

  ```json
  {
    "org_pubkey": "<32-byte ed25519 pubkey, 64hex>",
    "org_name": "acme",
    "version": 7,
    "members": [
      { "moss_peer_id": "<64hex>", "name": "alice", "role": "admin" },
      { "moss_peer_id": "<64hex>", "name": "bob",   "role": "member" }
    ],
    "sig": "<ed25519 signature over canonical bytes sans sig>"
  }
  ```

- Clients accept a roster iff the signature verifies against `org_pubkey`
  AND `version` is strictly greater than the stored one (anti-rollback).
- Member identity anchor is the **moss peer-id**: stable across restarts
  (persisted `moss_identity`, `persistence.rs`) and already used as the
  anti-spoof anchor on the relay path (`private_dm_runtime.rs` relay drain).
  MLS fingerprints are per-conversation and therefore not usable as durable
  identifiers.
- Note: today, invite-URI possession is the only gate on DMs; the displayed
  fingerprint is informational. The signed roster is therefore the first
  cryptographic membership gate in the app, not a replacement for an
  existing one.

## Components

### 1. Roster core (Rust, `src-tauri/src/adapters/org_roster.rs`)

Parse, canonicalize, verify, and diff rosters. Pure logic, no I/O.

- `verify(roster_bytes, expected_org_pubkey) -> Roster` — signature +
  version checks.
- `diff(old, new) -> {added, removed, role_changes}` — drives join/revoke
  actions.
- Canonical form: JCS-style canonical JSON (sorted keys, no whitespace).
  Signature covers the canonical bytes with `sig` removed.

### 2. Org runtime (Rust, `src-tauri/src/adapters/org_runtime.rs`)

One moss node per org (same lifecycle pattern as `channel_runtime.rs`).

- Org bundle URI: `mosh://org?mesh=<org_mesh_id>&name=<label>#org=<org_pubkey_64hex>`.
  The fragment carries the trust anchor; `mesh` is routing, same split as DM
  invites.
- Join flow (new member): join org mesh (existing infohash discovery from
  `mesh_id`), publish `OrgHello { moss_peer_id, display_name }` on
  `org-control/<org_mesh_id>`, re-published on the same resend cadence as DM
  key packages until acknowledged by a roster that includes the member.
- Roster distribution: latest roster is broadcast on `org-control/...` by
  any member that holds it (gossip; highest valid version wins), and offered
  to newly seen peers. Admin CLI injects new roster versions via its own
  mosh node joined to the same mesh.
- On roster update, emits events to the UI and to other runtimes:
  `member_added`, `member_removed`.

### 3. DM bootstrap from roster

Clicking a roster member opens/creates a DM without an invite URI:
generate `mesh_id`/`session_id` locally, send a directed
`OrgDmOffer { target_peer_id, mesh_id, session_id }` over org-control
(pattern already exists as `DmOffer` in `channel_runtime.rs`), both sides
enter the existing `create_invite`/`accept_invite` machinery. The offer is
accepted only if the sender's peer-id is in the current roster.

### 4. Revocation

Two layers, both driven by `diff().removed`:

- **Policy**: on receiving roster v+1 without member X, every client
  removes X's DM sessions/org-group membership from its UI, refuses new
  `OrgDmOffer`s from X, and stops relaying org traffic to X.
- **Crypto (admin-kick)**: in org private groups, the group admin issues an
  MLS Remove for X's leaf and commits. New capability in
  `mls_crypto.rs`: `remove_member(leaf_index)` — OpenMLS supports removing
  an arbitrary leaf; the app currently only removes `own_leaf_index`.
  Epoch advances; X cannot decrypt anything after the commit.

Revoked members keep their local history (unavoidable — data on their disk)
and can still read the org roster mesh traffic (roster contents are
membership metadata, not message content). Documented as a known property.

### 5. Membership resync (prerequisite for reliable revocation)

Known gap (`private_group_runtime.rs` comments): a member offline during a
commit desyncs permanently. Fix scoped to what revocation needs:

- Each group commit is stored (encrypted, `persistence.rs`) with its epoch
  by the admin and any member that observed it.
- `ResyncRequest { have_epoch }` / `ResyncResponse { commits[] }` control
  messages on the existing group control channel; on reconnect a member
  requests replay from its last epoch. Out-of-order commits are buffered
  keyed by epoch instead of dropped.
- If replay cannot bridge the gap (storage pruned), the member falls back
  to rejoin-via-roster (fresh KeyPackage, admin re-adds).

This fixes an existing correctness bug for all private groups, org or not.

### 6. Persistence

New redb tables in `persistence.rs`, following the `sessions` pattern:

- `org_roster`: single row per org — latest verified roster bytes.
- `group_commit_log`: `(group_id, epoch) -> commit bytes` for resync.

Both inside the existing AES-256-GCM envelope; no new key material in this
repo. The admin CLI keeps the org private key in its own storage.

### 7. Admin CLI (separate closed repo, out of scope here)

`mosh-org keygen | pending | approve | remove | bundle`. Consumes/produces
the roster format above; joins the org mesh via the same moss FFI to publish
signed rosters. This repo's only contract with it is the roster JSON format
and the bundle URI — documented in this spec, versioned by `version` field
semantics.

Nobody ever types a peer-id by hand. The flows are:

- **Member joins**: pastes the org bundle URI once (same gesture as a DM
  invite today). Their client publishes `OrgHello` automatically.
- **Admin approves**: `mosh-org pending` lists join requests as
  `<display_name>  <peer-id short hash>`; `mosh-org approve <name>` signs
  roster v+1 with that member's peer-id filled in from the hello. The short
  hash exists only for disambiguation when two hellos share a name.
- **Admin removes**: `mosh-org remove <name>`.
- Peer-ids appear in full only inside the roster document and logs.

## UI (minimal)

- "Join organization" input accepting the org bundle URI (next to existing
  invite input).
- Org sidebar section: member list from roster, click → DM.
- Removed-member state: conversation greyed with "no longer in <org>".

## Error handling

- Invalid signature / rollback version: reject silently, keep current
  roster, log at warn.
- Org mesh unreachable: roster is cached in redb; app works offline with
  the last known roster.
- `OrgDmOffer` from non-member: drop, log.
- Resync storage pruned: explicit "rejoin needed" state, not silent desync.

## Testing

- Roster core: pure unit tests — signature tamper, version rollback,
  canonicalization stability, diff correctness.
- `mls_crypto::remove_member`: cargo test — 3-party group, admin removes B,
  assert B's decrypt fails post-commit and A/C epochs advance.
- Resync: cargo test — member misses N commits, replays, converges;
  out-of-order buffer test.
- E2E (manual, 2 machines): bundle join, roster DM, revoke while target
  offline → target loses group access on reconnect.

## Build order

1. Roster core + persistence tables (pure, testable, no runtime wiring).
2. `mls_crypto::remove_member` (small, unblocks revocation).
3. Group commit log + resync path (fixes existing bug, largest risk item).
4. Org runtime: mesh, bundle URI, roster gossip, join flow.
5. DM bootstrap from roster + revocation wiring.
6. UI.
7. Admin CLI (separate repo, can start after step 1 freezes the format).
